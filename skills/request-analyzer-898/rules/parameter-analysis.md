# Parameter Analysis

Classify every query parameter, body field, cookie, and important header.

| Type | Examples | Main question |
|---|---|---|
| Fixed parameter | `version`, `platform`, `language`, `locale`, `app_name` | Can it be fixed safely? |
| Dynamic parameter | `timestamp`, `nonce`, `uuid`, `request_id`, `trace_id` | Can it be generated locally? |
| Session parameter | `cookie`, `session`, `token`, `Authorization`, `csrf` | Where does it come from and how long does it live? |
| Algorithm parameter | `sign`, `signature`, `encrypt`, `hash`, `x-bogus`, `checksum` | Is it validated by the server and can its generation logic be recovered? |
| Environment parameter | `device_id`, `browser_id`, `fingerprint`, `web_id`, `client_id` | Does it depend on browser, device, storage, or fingerprint state? |

## Classification Heuristics

- Fixed: stable across requests, version-like, locale-like, platform-like, or app metadata.
- Dynamic: timestamp-like, UUID-like, random-looking but locally generatable without server state.
- Session: login cookies, bearer tokens, CSRF tokens, server-issued identifiers, account-bound values.
- Algorithm: names or behavior suggest signing, encryption, HMAC, checksum, anti-tamper, request integrity, or replay prevention.
- Environment: values stored in localStorage/sessionStorage/indexedDB/cookies, generated by browser fingerprint code, device identity, canvas/WebGL/audio/browser entropy, or long-lived install identity.
- Unknown: insufficient evidence. Do not over-classify.

## Handling Decision Model

Do not assume every parameter needs reverse engineering. Decide the cheapest sufficient handling.

Preferred tests:

1. Delete test: remove the parameter and observe whether behavior changes.
2. Modify test: change the value and check whether the server validates it.
3. Fixed test: reuse a captured value and check lifetime and replay window.
4. Regeneration test: generate a local replacement and compare success.

Output one decision per parameter:

```text
Parameter:
Decision: no handling | simple generation | needs algorithm analysis | needs browser environment
Reason:
Required test:
Expected failure signal:
Implementation note:
Fallback:
```

Decision rules:

- Use `no handling` when delete/modify tests show no server impact or the value is display-only.
- Use `simple generation` for timestamps, UUIDs, nonce values, request IDs, and static metadata that can be generated locally.
- Use `needs algorithm analysis` only when evidence shows server validation or core interface behavior depends on an algorithm value.
- Use `needs browser environment` when the value depends on browser storage, page bootstrap state, WebSocket state, device fingerprint, or JS runtime APIs that are costly or unstable to emulate.
